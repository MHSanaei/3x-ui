# Self-steal для VLESS+Reality в 3X-UI: полное руководство

## Зачем это нужно

Протокол Reality (VLESS+Reality) прячет VPN-хендшейк внутри TLS-хендшейка настоящего сайта — стороннего "dest" (например, `www.amazon.com` или `www.apple.com`). Наблюдатель, который пытается достучаться до вашего сервера без правильного Reality-ключа, прозрачно перенаправляется на реальный внешний сайт и видит его настоящий контент.

Схема рабочая, но с изъяном: вы не владеете этим сайтом. Вы не контролируете, что именно увидит наблюдатель при более глубокой проверке, не знаете, когда чужой сертификат обновится неожиданным образом, и не можете увидеть в логах, кто именно и как часто пытается достучаться до вашего сервера без ключа — эти запросы физически обрабатывает чужой сервер, а не ваш.

**Self-steal** решает это, заменяя внешний dest на nginx, поднятый на этом же сервере. Технически это тот же принцип "воровства" TLS-идентичности, но теперь вы "воруете" у самого себя — то есть полностью контролируете:
- что видит любой, кто заходит на домен без Reality-ключа;
- какой сертификат используется и когда обновляется;
- логи всех попыток подключения — с реальными IP, а не единственным `127.0.0.1`.

## Предпосылки

Это руководство предполагает, что у вас уже есть:
- 3X-UI, установленный и работающий;
- рабочий VLESS+Reality-инбаунд на TCP:443 с обычным внешним dest;
- домен, привязанный к серверу, и уже выпущенный сертификат Let's Encrypt (обычно acme.sh ставится вместе с панелью);
- root-доступ по SSH.

## Технические требования к dest — это не совет, а протокол

Сам механизм Reality не принимает произвольный сайт в качестве dest. В исходниках панели (`internal/web/service/reality_scan.go`, функция `probeRealityAddr`) есть буквально такая проверка:

```go
res.Feasible = res.TLS13 && res.H2 && res.X25519 && res.CertValid
```

Dest обязан одновременно отвечать: **TLS 1.3**, **HTTP/2** (ALPN `h2`), key exchange на **X25519**, и иметь **валидный сертификат**, у которого домен совпадает с тем, что запрашивается. Причина не капризная: Reality прячет свою аутентификационную подпись именно в полях TLS 1.3-хендшейка, использующих X25519 — если этой комбинации нет, механизму физически некуда встроить подпись. Это значит, что ваш собственный nginx должен соответствовать этим же требованиям — обычная установка "из коробки" этого не гарантирует.

## Шаг 1 — Ставим nginx из официального репозитория

Пакет nginx в стандартных репозиториях Ubuntu обычно отстаёт на пару стабильных релизов от актуального nginx.org. Для Reality лучше брать самую свежую сборку:

```bash
apt install -y curl gnupg2 ca-certificates lsb-release ubuntu-keyring
curl -s https://nginx.org/keys/nginx_signing.key | gpg --dearmor | tee /usr/share/keyrings/nginx-archive-keyring.gpg >/dev/null
echo "deb [signed-by=/usr/share/keyrings/nginx-archive-keyring.gpg] http://nginx.org/packages/ubuntu $(lsb_release -cs) nginx" > /etc/apt/sources.list.d/nginx.list
printf 'Package: *\nPin: origin nginx.org\nPin-Priority: 900\n' > /etc/apt/preferences.d/99nginx
apt update
apt install -y nginx
```

Проверить, что кодовое имя вашего дистрибутива (`lsb_release -cs`) вообще поддерживается nginx.org, можно так: `curl -s http://nginx.org/packages/ubuntu/dists/ | grep -oE 'href="[a-z]+/"'`.

> **Ловушка:** если на сервере остался "хвост" от старого дистрибутивного пакета nginx (например, `nginx-common` в статусе `rc` — установлен ранее, потом удалён не полностью), **не удаляйте его отдельной командой**. `apt purge nginx-common` в такой ситуации может снести `/etc/nginx/nginx.conf`, `/etc/nginx/conf.d/default.conf` и весь `/var/log/nginx/`, если dpkg по ошибке считает эти файлы принадлежащими старому пакету. Если это уже произошло — спасает полный цикл `apt purge nginx && apt install nginx` (не `--reinstall`: у него отдельный баг, из-за которого conffiles не восстанавливаются, если физически удалены, а в базе dpkg числятся как "не изменённые").

## Шаг 2 — Готовим правдоподобный decoy-сайт

Это критически важный и часто недооценённый шаг. **Не используйте страницу-заглушку nginx по умолчанию** ("Welcome to nginx!") — это моментально выдаёт, что сервер настроен наспех именно для сокрытия чего-то, а не для реального бизнеса.

Нужен полноценный лендинг:
- реальный текст, не lorem ipsum и не заглушка;
- минимум 2-3 внутренние страницы с реальным содержанием (`/privacy`, `/terms`) — ссылки в никуда (`href="#"`) заметны при ручном осмотре и тоже являются "tell";
- `robots.txt`;
- нейтральная тематика, никак не связанная с вашей настоящей инфраструктурой. Контактный email на decoy-сайте **не должен** совпадать по названию с доменом/проектом VPN — иначе появляется прямая корреляция между "обычным сайтом" и вашей VPN-инфраструктурой при простом поиске.

Разместите сайт в `/var/www/<node>-decoy/`. Для чистых URL без `.html` создавайте структуру вида `/privacy/index.html`, `/terms/index.html` — nginx автоматически резолвит `/privacy` → редирект → `/privacy/` → отдаёт `index.html` внутри директории, через стандартную связку `index` + `try_files`.

## Шаг 3 — Конфиг nginx

Создайте `/etc/nginx/conf.d/<node>-selfsteal.conf`:

```nginx
# Обычный HTTP → HTTPS редирект — то, что делает практически любой настоящий
# сайт. Обязателен с самого начала: официальный пакет nginx.org ставит
# собственный conf.d/default.conf с "listen 80; server_name localhost;".
# Если не добавить сюда отдельный блок под ваш домен, ЭТОТ default.conf
# останется единственным слушателем порта 80 — а значит абсолютно весь
# HTTP-трафик, включая запросы к вашему домену, будет попадать именно туда
# и получать голую заглушку "Welcome to nginx!" вместо вашего decoy-сайта.
# Прямое несоответствие: на 443 — полноценный сайт, на 80 — "сервер только
# что установлен, конфигурация не завершена". Ровно то, что выдаёт неладное
# при случайном или намеренном осмотре.
server {
    listen 80;
    server_name <DOMAIN>;
    return 301 https://$host$request_uri;
}

server {
    listen 127.0.0.1:8443 ssl proxy_protocol;
    http2 on;
    server_name <DOMAIN>;

    # Xray форвардит сюда "украденные" (неаутентифицированные) соединения
    # с флагом xver=1 (PROXY protocol v1). Без этой связки $remote_addr
    # всегда будет 127.0.0.1 — то есть сам Xray, а не реальный клиент.
    set_real_ip_from 127.0.0.1;
    real_ip_header proxy_protocol;

    ssl_certificate     /root/cert/<DOMAIN>/fullchain.pem;
    ssl_certificate_key /root/cert/<DOMAIN>/privkey.pem;

    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ecdh_curve X25519:secp256r1;
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 1d;

    access_log /var/log/nginx/<node>-decoy-access.log;
    error_log  /var/log/nginx/<node>-decoy-error.log;

    # Реальный порт nginx (8443) — внутренний. Он не должен утечь в
    # автоматически генерируемые редиректы (например, стандартный переход
    # /privacy -> /privacy/ по умолчанию собирает абсолютный Location
    # с портом сервера — это выдаёт нестандартную схему проксирования).
    absolute_redirect off;

    root /var/www/<node>-decoy;
    index index.html;

    location / {
        try_files $uri $uri/ =404;
    }
}
```

Обратите внимание: nginx слушает **127.0.0.1:8443**, а не публичный 443 напрямую. Публичный порт 443 по-прежнему принадлежит Xray — именно Xray решает на основе Reality-хендшейка, форвардить ли соединение на этот внутренний адрес (нелегитимные клиенты) или обслуживать его как настоящий VLESS (легитимные клиенты с верным ключом).

**Про сертификат:** переиспользуйте уже существующий сертификат домена, выпущенный через acme.sh (тот же, что использует панель или Hysteria2-инбаунд) — не заказывайте отдельный, это один и тот же домен.

Проверка и запуск:
```bash
nginx -t && systemctl enable --now nginx
```

## Шаг 4 — Перевыпуск сертификата должен перезагружать и nginx

По умолчанию acme.sh при продлении сертификата перезапускает только саму панель (x-ui). Нужно расширить hook, чтобы он также перезагружал nginx:

```bash
~/.acme.sh/acme.sh --install-cert -d <DOMAIN> --ecc \
  --key-file       /root/cert/<DOMAIN>/privkey.pem \
  --fullchain-file /root/cert/<DOMAIN>/fullchain.pem \
  --reloadcmd 'systemctl restart x-ui || rc-service x-ui restart; systemctl reload nginx'
```

Проверить, что записалось: `cat ~/.acme.sh/<DOMAIN>_ecc/<DOMAIN>.conf | grep reloadcmd` (значение хранится в base64, декодировать `base64 -d`).

## Шаг 5 — Настройка инбаунда через интерфейс панели

Всё нужное есть прямо в форме — лезть в API или тем более в SQLite напрямую не требуется, это обычная правка через UI.

Входящие → нужный VLESS-инбаунд → вкладка «Безопасность»:
- **Цель** (Target) → `127.0.0.1:8443`;
- **SNI** → только ваш домен (стереть весь список чужих доменов, которые были там при внешнем dest);
- **Xver** → `1` (включает PROXY protocol v1 — без этого nginx не увидит реальный IP клиента, только `127.0.0.1`);
- если раньше генерировали mldsa65 (кнопка «Получить новый Seed» делает это) — сразу очистить оба поля `mldsa65 Seed` / `mldsa65 Verify` (см. известный баг ниже).

Сохранить. **Xray перезапускать вручную не нужно** — у панели есть собственный фоновый job, который проверяет каждые 30 секунд (`@every 30s`, `internal/web/web.go`), не выставлен ли где-то флаг "нужен рестарт", и сам перезапускает Xray после любого изменения инбаунда. Через полминуты можно проверять живой конфиг.

Проверить, что изменения реально применились (не только сохранились в форме): на сервере прочитать `/usr/local/x-ui/bin/config.json` и сверить `xver`/`target`/`serverNames`.

> **Если провижините ноды скриптами, а не руками** — тот же результат достижим через API (Bearer-токен: Settings → Security → API Token, либо уже в `/etc/x-ui/install-result.env`): `GET /panel/api/inbounds/list` → поправить `streamSettings.realitySettings.target`/`serverNames`/`xver` в теле → `POST /panel/api/inbounds/update/:id` (апдейт заменяет объект целиком, частичный патч не поддерживается). Тот же фоновый job подхватит изменения и здесь; форсировать `POST /panel/api/server/restartXrayService` отдельно нужно, только если не хотите ждать до 30 секунд — например, при автоматизированной проверке сразу после апдейта.

## Известный баг: mldsa65 ломает реальные подключения

**Симптом:** настоящий VLESS+Reality клиент не может подключиться — соединение обрывается по EOF после нескольких ретраев (`common/retry: [EOF] > common/retry: all retry attempts failed`), при этом fallback-путь (обычный браузер без Reality-ключа) работает нормально и получает decoy-сайт.

**Причина:** задокументированный баг Xray-core ([issue #5319](https://github.com/XTLS/Xray-core/issues/5319), закрыт мейнтейнерами как "not planned"). Если у инбаунда заполнены поля `mldsa65Seed` (серверная часть, пост-квантовая подпись) и соответствующий `mldsa65Verify` на клиенте, реальное соединение ломается — независимо от того, насколько правильно оба значения сгенерированы и синхронизированы между собой.

**Фикс:** на той же вкладке «Безопасность», прямо под полями `mldsa65 Seed` / `mldsa65 Verify`, есть кнопка **«Очистить»** — жмёте её и сохраняете инбаунд, оба поля станут пустыми. Не пытайтесь настроить mldsa65 "правильно" — это заведомо не работает на данный момент. Если панель уже сгенерировала эти поля сама при создании инбаунда (кнопка «Получить новый seed» делает это автоматически) — сразу очищайте их, не дожидаясь проблемы на практике.

> Для скриптовой автоматизации то же самое — через API-апдейт того же инбаунда:
> ```python
> realitySettings['mldsa65Seed'] = ''
> realitySettings['settings']['mldsa65Verify'] = ''
> ```

## Проверка — обязательно живыми запросами

Не полагайтесь на то, что "конфиг выглядит правильно". Проверяйте вживую.

### 1. Fallback-путь (без Reality-ключа), с любого внешнего хоста

```bash
# Параметры TLS
echo | openssl s_client -connect <IP>:443 -servername <DOMAIN> -tls1_3 2>&1 | grep -E 'subject=|issuer=|Protocol|Verify return'
echo | openssl s_client -connect <IP>:443 -servername <DOMAIN> -tls1_3 2>&1 | grep -i "temp key"   # должен быть X25519

# Контент
curl -sk --resolve <DOMAIN>:443:<IP> https://<DOMAIN>/ -o /dev/null -w 'HTTP: %{http_code}, version: %{http_version}\n'
```

Ожидаемый результат: TLS 1.3, X25519, валидный сертификат на нужный домен, HTTP/2, код ответа 200.

### 2. Реальный IP пробующего клиента виден в логах decoy-сайта

После настройки `xver` + `proxy_protocol` отправьте внешний запрос и проверьте `tail /var/log/nginx/<node>-decoy-access.log` — там должен быть настоящий IP клиента, а не `127.0.0.1`.

### 3. Настоящее VLESS+Reality-подключение (не только fallback)

Для теста берите **тот же бинарник xray, что использует сама панель** (`scp root@<node>:/usr/local/x-ui/bin/xray-linux-amd64`), а не последний релиз с GitHub — версии могут расходиться, и одно это способно дать ложный EOF, не имеющий отношения к реальной проблеме конфигурации.

Создайте временного тестового клиента через интерфейс: Клиенты → «Добавить клиента» → выбрать нужный VLESS-инбаунд, Flow → `xtls-rprx-vision`, email, например, `selfsteal-test-temp`. Сохранить.

Не собирайте параметры подключения (publicKey, shortId) вручную из базы или API-ответа — у только что созданного клиента в списке есть кнопка с QR-кодом/ссылкой, которая уже формирует полностью готовую и корректную `vless://`-ссылку сама. Возьмите нужные значения оттуда (или экспортируйте ссылку целиком и импортируйте в любой VLESS-совместимый клиент).

Дальше нужен отдельный запущенный xray-клиент с этими параметрами — `vless`-outbound, `security: reality`, `serverName` = ваш домен, `fingerprint: firefox`, `flow: xtls-rprx-vision`, **без `mldsa65Verify`**. Направьте через него SOCKS5 на `127.0.0.1:10808` и проверьте:
```bash
curl --socks5-hostname 127.0.0.1:10808 https://ifconfig.me
```
Ответ должен быть IP-адресом самой ноды, а не вашим собственным.

**После теста удалите временного клиента** — там же, в списке клиентов, обычной кнопкой удаления.

## Итоговый чек-лист

- [ ] nginx установлен из nginx.org, не из дистрибутивного репозитория
- [ ] decoy-сайт — полноценный, с реальным текстом, минимум двумя внутренними страницами и `robots.txt`; контактный email не связан по названию с VPN-инфраструктурой
- [ ] nginx слушает `127.0.0.1:8443`, публичный 443 остаётся за Xray
- [ ] отдельный блок `listen 80` с 301-редиректом на https добавлен — без него порт 80 отдаёт заглушку "Welcome to nginx!" от стокового `default.conf`
- [ ] `absolute_redirect off` — иначе внутренний порт 8443 утечёт в редиректы
- [ ] сертификат переиспользован, а не выпущен отдельно; `--reloadcmd` acme.sh дополнен перезагрузкой nginx
- [ ] в панели: «Цель» = `127.0.0.1:8443`, SNI = только ваш домен, Xver = 1
- [ ] изменения сверены в живом `config.json` (фоновый job подхватывает их сам в течение 30 секунд, форсировать рестарт вручную не обязательно)
- [ ] `mldsa65Seed`/`mldsa65Verify` очищены (или не заполнялись вовсе)
- [ ] fallback-путь проверен живыми запросами (TLS 1.3, X25519, валидный сертификат, 200)
- [ ] реальный IP клиента виден в логах decoy-сайта, не `127.0.0.1`
- [ ] настоящее VLESS+Reality-подключение проверено тестовым клиентом (тем же бинарником xray, что у панели), тестовый клиент удалён после проверки
